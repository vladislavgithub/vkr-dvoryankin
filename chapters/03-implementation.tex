\section{Проектирование и реализация сервиса}
\thispagestyle{plain}

В настоящей главе описана архитектура сервиса EduQuiz, обоснован
технологический стек, представлена структура базы данных
и API. Отдельно рассмотрены вопросы интеграции с университетской
LMS \textit{edu.gubkin} и развёртывания в инфраструктуре Российской
Федерации.

\subsection{Архитектура клиент-сервер}

Архитектура сервиса --- классическая трёхуровневая клиент-серверная
с собственным backend-компонентом, что концептуально отличает её
от serverless-варианта в предшествующей научно-исследовательской
работе. На уровне контейнерной диаграммы C4 выделяются:
\begin{itemize}[noitemsep]
    \item кроссплатформенный клиент (Flutter), доступный на Web,
          iOS, Android и macOS;
    \item REST/WebSocket API на языке Go (Gin), отвечающий за
          бизнес-логику, аутентификацию и трансляцию событий
          в реальном времени;
    \item PostgreSQL --- единственный источник истины
          для пользовательских данных, курсов, банков вопросов,
          сессий и состояний геймификации;
    \item Redis --- кеш, хранилище refresh-токенов и шина
          событий (pub/sub) между инстансами API;
    \item обратный прокси Caddy с автоматическим выпуском TLS,
          выступающий точкой входа из публичной сети.
\end{itemize}

Все компоненты, кроме клиента и Caddy, скрыты за внутренней сетью.
Между сервисами установлено только то взаимодействие, которое
требуется бизнес-логикой (рис.~\ref{fig:c4-containers}).

\subsection{Технологический стек}

\textbf{Frontend.} Flutter 3.35 с архитектурой feature-first.
Управление состоянием --- \textit{Riverpod 2.6} с провайдерами,
сгенерированными через \texttt{riverpod\_generator}.
Маршрутизация --- \texttt{go\_router 14}; декларативная схема
маршрутов с защитой по роли пользователя.
Стиль интерфейса --- Material Design 3 с темой, построенной
от seed-цвета корпоративной палитры РГУ Нефти и Газа.
Адаптивная вёрстка обеспечивает корректное отображение
на устройствах от 320~px (мобильный портрет) до 4К (десктоп).
HTTP-клиент --- \texttt{dio} с типизированным API через \texttt{retrofit};
real-time канал --- \texttt{web\_socket\_channel}; модели
описываются с помощью \texttt{freezed} и \texttt{json\_serializable}.
Анимации --- \texttt{flutter\_animate} и \texttt{lottie};
графики дашбордов --- \texttt{fl\_chart}.

\textbf{Backend.} Go 1.25 с фреймворком \textit{Gin}.
WebSocket-хаб реализован на базе \texttt{gorilla/websocket}.
Доступ к PostgreSQL --- через драйвер \texttt{pgx/v5}; миграции
управляются \texttt{golang-migrate}. Аутентификация ---
JWT-токены, выпускаемые сервисом без зависимости от внешних
провайдеров идентификации.

\textbf{СУБД и инфраструктура.} PostgreSQL~16 с включёнными
расширениями \texttt{citext} и \texttt{pgcrypto};
Redis~7; Caddy с автоматическим выпуском TLS-сертификата.
Контейнеризация --- Docker и Docker Compose; одна composition
описывает как development-, так и production-среду
(с подменой переменных окружения).

\subsection{Структура базы данных}

База данных спроектирована как нормализованная (3НФ) реляционная
схема. Базовые сущности и связи приведены ниже; полные определения
содержатся в файлах миграций \texttt{migrations/0001\_init} ---
\texttt{migrations/0004\_gamification}.

\textbf{Пользователи и роли.}
\begin{lstlisting}[language=SQL]
CREATE TYPE user_role AS ENUM ('teacher', 'student', 'admin');

CREATE TABLE users (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email         CITEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    full_name     TEXT NOT NULL,
    role          user_role NOT NULL DEFAULT 'student',
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
\end{lstlisting}

\textbf{Курсы и банки вопросов} (миграция 0002).
Курс принадлежит преподавателю; банк вопросов привязан к курсу;
вопросы хранят варианты в JSONB, что обеспечивает гибкость
без необходимости менять схему при появлении новых типов.
Тип вопроса описывается ENUM \texttt{question\_kind}
(\texttt{single\_choice}, \texttt{multi\_choice}, \texttt{open\_text},
\texttt{rating}, \texttt{qna}).

\textbf{Интерактивные сессии и ответы} (миграция 0003).
Таблица \texttt{rooms} хранит состояние комнаты, шестизначный
код доступа, текущий вопрос, очередь и список заданных вопросов.
Уникальное ограничение
\texttt{UNIQUE (room\_id, question\_id, participant\_id)}
на таблице \texttt{answers} предотвращает повторные отправки
одного и того же ответа. Конечный автомат состояния комнаты
реализован через ENUM \texttt{room\_status}.

\textbf{Геймификация} (миграция 0004). Журнал начислений
очков (\texttt{xp\_log}), денормализованный агрегат
\texttt{user\_progress} с текущим XP, уровнем и серией дней,
справочник бейджей (\texttt{badges}, заполнен seed-данными),
и состояние SM\textendash 2 для пары «пользователь, вопрос»
с полями \texttt{ef}, \texttt{interval\_days}, \texttt{repetitions},
\texttt{due\_date}.

\subsection{REST API и WebSocket-протокол}

\textbf{REST API} соответствует спецификации OpenAPI 3.0.
Семейство эндпоинтов:
\begin{itemize}[noitemsep]
    \item \texttt{POST /api/v1/auth/register},
          \texttt{POST /api/v1/auth/login},
          \texttt{GET  /api/v1/auth/me};
    \item \texttt{GET/POST/PATCH/DELETE /api/v1/courses};
    \item \texttt{POST /api/v1/courses/\{id\}/banks/import}
          --- импорт банка вопросов из LMS;
    \item \texttt{POST /api/v1/rooms} --- создание комнаты,
          возвращает шестизначный код;
    \item \texttt{POST /api/v1/rooms/\{id\}/start},
          \texttt{POST /api/v1/rooms/\{id\}/next};
    \item \texttt{POST /api/v1/rooms/join} --- студент подключается
          по коду;
    \item \texttt{POST /api/v1/rooms/\{id\}/answers} --- отправка
          ответа.
\end{itemize}

\textbf{WebSocket-протокол.} Клиент устанавливает соединение
с эндпоинтом \texttt{/ws/rooms/\{id\}} и аутентифицируется
JWT-токеном через query-параметр (для совместимости с браузерами).
Все события передаются в формате JSON с обязательными полями
\texttt{type} и \texttt{data}. Основные типы событий:
\begin{itemize}[noitemsep]
    \item \texttt{room.state\_changed} --- изменение статуса комнаты;
    \item \texttt{question.activated} --- сервер активировал новый
          вопрос (от сервера к клиенту);
    \item \texttt{question.answered} --- студент отправил ответ
          (преподавателю в реальном времени);
    \item \texttt{leaderboard.updated} --- обновление таблицы
          лидеров после ответа.
\end{itemize}

Внутренняя реализация: WebSocket-хаб подписан на Redis pub/sub,
что позволяет горизонтально масштабировать API в нескольких
инстансах без потери real-time-канала между участниками сессии.

\subsection{Интеграция с LMS edu.gubkin}

Поддерживаются следующие способы импорта банка вопросов:
\begin{enumerate}[label=\arabic*., noitemsep]
    \item \textbf{Импорт файла Moodle XML} --- стандартный экспорт
          платформы Moodle, поддерживаемый в LMS edu.gubkin.
          Обработчик разбирает XML и создаёт записи
          в \texttt{question\_banks} и \texttt{questions}, маппя
          типы вопросов Moodle на ENUM \texttt{question\_kind}.
    \item \textbf{Импорт формата GIFT} --- текстовый формат,
          удобный для быстрого ввода преподавателем.
    \item \textbf{Импорт CSV} --- для случаев, когда вопросы
          хранятся в табличной форме.
    \item \textbf{API-интеграция} --- при наличии API
          edu.gubkin синхронизация выполняется автоматически
          с обновлением банка вопросов по расписанию (вопрос
          уточняется с эксплуатацией портала).
\end{enumerate}

Интеграция реализована как набор импортёров, объединённых
общим интерфейсом \texttt{Importer}. Преподаватель в кабинете
выбирает источник, загружает файл (или выбирает курс edu.gubkin),
просматривает предполагаемое количество и типы вопросов
и подтверждает импорт.

\subsection{Реализация клиента и серверной части}

\textbf{Структура проекта Flutter}
(репозиторий \texttt{eduquiz-flutter}):
\begin{lstlisting}
lib/
  main.dart           # ProviderScope -> EduQuizApp
  app.dart            # MaterialApp.router + Material 3
  core/
    theme/            # AppTheme: light/dark, seed-color
    router/           # go_router конфигурация
    api/              # dio-провайдер, baseUrl через --dart-define
  features/
    auth/             # вход, регистрация
    teacher/          # кабинет преподавателя
    student/          # кабинет студента
    gamification/     # XP, бейджи, лидерборд, прогресс
    repetition/       # SM-2 экраны повторения
  data/
    api/              # типизированные клиенты (retrofit)
    models/           # freezed-модели
\end{lstlisting}

\textbf{Структура backend} (репозиторий \texttt{eduquiz-backend}):
\begin{lstlisting}
cmd/api/main.go       # точка входа, graceful shutdown
internal/
  config/             # env-конфигурация
  server/             # Gin-сервер, healthz/readyz
  handlers/           # REST + WS обработчики
  services/           # бизнес-логика (XP, SM-2, scoring)

  ws/                 # WebSocket-хаб

migrations/           # SQL-миграции (golang-migrate)
docker-compose.yml    # postgres + redis + api
Dockerfile            # multi-stage, distroless, nonroot
\end{lstlisting}

\subsection{Развёртывание}

В development-среде предусмотрено развёртывание двумя способами:
\begin{itemize}[noitemsep]
    \item \texttt{make dev} --- запускает PostgreSQL и Redis
          в Docker-контейнерах, а API --- локально через
          \texttt{go run};
    \item \texttt{make up} --- поднимает все сервисы (включая API)
          в контейнерах одной командой.
\end{itemize}

В production-среде сервис разворачивается на VPS Selectel/Timeweb
(2 vCPU, 4 ГБ RAM, 40 ГБ SSD, ОС Ubuntu Server LTS). Деплой
автоматизирован GitHub Actions: на тег вида \texttt{v*.*.*}
собирается Docker-образ, отправляется в GitHub Container Registry
(GHCR), после чего на сервере выполняется \texttt{docker compose pull \&\& up -d}
по защищённому каналу SSH. Caddy автоматически выпускает
TLS-сертификат при первом запуске.
