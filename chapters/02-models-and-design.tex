\subsection{Архитектурное решение и стек}

\subsubsection{Выбор архитектурного стиля}

Согласно требованиям, сформулированным в подразделе~1.3.3, сервис
должен размещаться в инфраструктуре, контролируемой образовательной
организацией, и предоставлять открытый программный интерфейс
для интеграции с внешними системами. Эти требования определяют
необходимость собственного серверного приложения, развёртываемого
в инфраструктуре, доступной образовательной организации.

В качестве альтернативных архитектурных моделей рассматривались
монолитная и микросервисная архитектуры~\cite{fowler_poeaa,kleppmann2017}.

\emph{Монолитная архитектура} с единым серверным приложением,
обрабатывающим все запросы. Преимущества~--- простота разработки
и развёртывания, минимальные накладные расходы на сетевое
взаимодействие. Недостатки~--- ограниченная возможность
независимого масштабирования отдельных подсистем.

\emph{Микросервисная архитектура} с декомпозицией по бизнес-доменам.
Преимущества~--- независимое масштабирование, возможность применения
различных технологий в разных сервисах. Недостатки~--- высокая
операционная сложность, избыточность для прогнозируемого профиля
нагрузки учебной аудитории одного университета.

С учётом профиля нагрузки разрабатываемого сервиса (десятки
параллельных учебных сессий по 30--500 участников каждая) выбрана
монолитная архитектура с собственным backend-приложением.
Экспериментальная оценка производительности при $N = 500$
одновременных обучающихся (см.~подраздел~3.7) подтверждает, что
монолитная конфигурация обеспечивает нефункциональное требование
производительности; накладные расходы микросервисной декомпозиции
на сетевое взаимодействие и эксплуатацию в рассматриваемом профиле
нагрузки не окупаются. Описание
архитектуры приведено в нотации модели C4, обеспечивающей
визуализацию системы на четырёх уровнях детализации~\cite{brown_c4}.

\subsubsection{Архитектура клиент-серверного решения}

Архитектура разработанного сервиса представлена
на Рисунке~\ref{fig:c4-containers} в виде контейнерной диаграммы
C4. В состав системы входят следующие контейнеры.

\image{c4-containers.png}{Контейнерная диаграмма C4 сервиса EduQuiz}{fig:c4-containers}{0.95}

\emph{Клиентское приложение} (Flutter) функционирует на устройствах
преподавателя и обучающихся в четырёх вариантах сборки~--- Web, iOS,
Android, macOS.

\emph{Серверное приложение} (Go, Gin) реализует бизнес-логику,
аутентификацию и трансляцию событий интерактивных сессий через
WebSocket-канал.

\emph{Сервер базы данных} (PostgreSQL~16) служит единственным
источником истины для пользовательских данных, курсов, банков
вопросов, состояния сессий и состояний игровой прогрессии.

\emph{Сервер кеширования} (Redis~7) используется для хранения
refresh-токенов и в качестве шины событий между экземплярами
серверного приложения по протоколу publish/subscribe.

\emph{Обратный прокси-сервер} (Caddy~2) выступает точкой входа
из публичной сети, обеспечивает терминацию TLS и автоматический
выпуск сертификата по протоколу ACME.

Дополнительно на контейнерной диаграмме показаны компоненты
системы мониторинга (Prometheus, Grafana) и внешняя информационная
система университета (edu.gubkin)~--- они отнесены к перспективным
компонентам и реализуются на последующих этапах работы
(см.~подраздел~3.8).

Порты сервера базы данных и сервера кеширования не публикуются
наружу: взаимодействие выполняется только в пределах внутренней
сети контейнерного окружения, что снижает поверхность атаки
в соответствии с принципом эшелонированной защиты~\cite{owasp_asvs}.

\subsubsection{Выбор технического стека}

\emph{Серверная часть.} В качестве языка программирования серверной
части выбран язык Go версии 1.25. В качестве альтернатив
рассматривались Python с фреймворком FastAPI и Node.js
с фреймворком NestJS. Выбор Go обусловлен сочетанием статической
типизации (обнаружение значительной части ошибок на этапе
компиляции), встроенной модели конкурентности на основе горутин
и каналов (удобной для сценариев с большим числом одновременных
сетевых соединений) и низкого расхода оперативной памяти на одно
соединение, что важно при поддержке десятков WebSocket-соединений
в рамках одной учебной сессии. Python с интерпретатором CPython
не обеспечивает истинного параллельного выполнения потоков
вследствие глобальной блокировки интерпретатора (GIL); Node.js
обеспечивает асинхронную обработку запросов, но не предоставляет
статической типизации без перехода на язык TypeScript.
В качестве веб-фреймворка использован Gin~\cite{gin_docs}; для
работы с базой данных~--- драйвер jackc/pgx/v5; для миграций
схемы~--- утилита golang-migrate; для поддержки WebSocket~---
библиотека gorilla/websocket.

\emph{Реляционная база данных.} В качестве СУБД выбрана
PostgreSQL версии 16~\cite{postgresql_docs}. Выбор обусловлен
открытой лицензией, поддержкой типа JSONB (бинарный формат
хранения JavaScript Object Notation, JSON) для хранения вариантов
ответов на вопросы без изменения схемы при добавлении новых типов,
наличием расширений citext (регистронезависимые электронные
адреса) и pgcrypto (генерация универсально-уникальных идентификаторов
Universally Unique Identifier, UUID, и хеширование).

\emph{Кеширование и шина событий.} В качестве средства кеширования
и брокера событий между экземплярами серверного приложения
использована Redis версии 7. Обратный прокси Caddy
версии 2~\cite{caddy_docs} обеспечивает автоматический выпуск
TLS-сертификатов по протоколу ACME.

\emph{Клиентская часть.} В качестве платформы разработки
клиентского приложения выбран Flutter версии 3.35~\cite{flutter_docs}
с языком программирования Dart. Выбор Flutter обоснован
возможностью разработки единой кодовой базы, разворачиваемой
на четырёх целевых платформах (Web, iOS, Android, macOS) без
существенной потери качества пользовательского интерфейса.
Управление состоянием приложения выполнено с использованием
библиотеки Riverpod~2.6~\cite{riverpod_docs}; маршрутизация~---
средствами go\_router~14 с декларативной защитой маршрутов
по роли пользователя.

\emph{Контейнеризация и развёртывание.} Серверные компоненты
упакованы в Docker-контейнеры~\cite{docker_docs}. Композиция
описана в файле docker-compose.yml, охватывающем
как среду разработки, так и продуктивную среду, что обеспечивает
воспроизводимость развёртывания.

\subsection{Механизмы аутентификации и авторизации}

\subsubsection{Ролевая модель пользователей и разграничение доступа}

В разработанном сервисе выделено три роли пользователей:
преподаватель, обучающийся, администратор. Распределение прав
по ролям определяется по принципу наименьших
привилегий~\cite{owasp_asvs,gost58833}: каждая роль обладает
минимальным набором прав, необходимым для выполнения целевых
сценариев.

\emph{Преподаватель} создаёт курсы, формирует и редактирует банки
вопросов, проводит интерактивные учебные сессии и получает
аналитические отчёты по результатам сессий.

\emph{Обучающийся} подключается к сессии по числовому коду,
отвечает на вопросы, выполняет индивидуальное повторение учебного
материала по алгоритму SM-2 и получает доступ к собственной
статистике.

\emph{Администратор} обладает полным доступом к функциональности
преподавателя и обучающегося, управляет учётными записями
и правами доступа к курсам.

Связь ролей с конкретными разрешёнными операциями реализуется
через гибридную модель Role-Based Access Control (RBAC)
с дополнительной проверкой принадлежности ресурса по атрибуту
владения (owner\_id). Базовая часть модели~--- RBAC,
закрепляющий за каждой ролью фиксированный набор прав на чтение,
запись и изменение ресурсов. Атрибутное расширение~--- сверка
идентификатора владельца ресурса в каждом обработчике защищённого
эндпоинта с идентификатором запрашивающего пользователя, что
обеспечивает изоляцию данных между преподавателями. В практике
разграничения доступа известны и другие модели~--- дискреционная
(DAC), мандатная (MAC), атрибутная (ABAC) и реляционная (ReBAC).
Сопоставление моделей по применимости к задаче разрабатываемого
сервиса приведено в табл.~\ref{tab:access-models}.

\subsubsection{Реализация JWT-аутентификации}

Аутентификация пользователей реализована на основе технологии
JSON Web Token~\cite{rfc7519}. После успешной проверки пары
«логин~--- пароль» сервис выдаёт пару токенов: токен доступа
со сроком действия 15 минут и токен обновления со сроком действия
7 суток. Подпись токенов выполняется алгоритмом HS256 (HMAC SHA-256)
с серверным секретом. В монолитной архитектуре с единым доверенным
контуром симметричный HS256 устраняет необходимость управления
парой открытого и закрытого ключей, требуемой для асимметричного
RS256. Токен
доступа содержит идентификатор пользователя и его роль; проверка
подписи токена выполняется локально каждым обработчиком запроса
без обращения к серверу аутентификации, что устраняет необходимость
хранения серверной сессии.

Хеширование паролей выполнено алгоритмом bcrypt с рабочим
параметром $\text{cost} = 11$, что обеспечивает время вычисления
хеша порядка 100~мс на стенде, описанном в подразделе~3.7.
Значение параметра соответствует рекомендации OWASP Password
Storage Cheat Sheet 2024 (work factor 10--12) и представляет собой
компромисс между скоростью обработки запроса и стойкостью к атакам
методом полного перебора~\cite{owasp_asvs}.

Для предотвращения атак методом полного перебора перед эндпоинтами
аутентификации установлен ограничитель скорости запросов
по IP-адресу (token-bucket, не более 20 запросов в минуту).
Рефреш-токены ротируются при каждом обновлении и проверяются
на серверной стороне по таблице сессий, что снижает риск
компрометации при перехвате токена.

Выбор JWT в качестве схемы аутентификации обусловлен сопоставлением
с распространёнными альтернативами, приведённым
в табл.~\ref{tab:auth-schemes}. Существенным аргументом
в пользу JWT по сравнению со схемой Session cookie с разделяемым
хранилищем сессий в Redis является исключение сетевого обращения
в hot-path обработки запроса: подпись токена доступа проверяется
обработчиком локально по серверному секрету без обращения к Redis,
что снижает латентность критического пути по сравнению с проверкой
идентификатора сессии в разделяемом хранилище на каждый запрос.
Дополнительно при подключении WebSocket~\cite{rfc6455} из браузера
передача cookie между источниками ограничена атрибутом SameSite
и не во всех конфигурациях разрешена, тогда как подписанный токен
JWT передаётся в параметре строки запроса единообразно. Полностью
stateless-схему предложенный вариант не образует: рефреш-токены
хранятся в таблице сессий для возможности отзыва и ротации,
поэтому реализованная схема является гибридной~--- stateless
по токену доступа и stateful по токену обновления.

\subsubsection{Защита от типовых угроз}

Защита сервиса спроектирована в соответствии с требованиями
ГОСТ Р 56939-2024~\cite{gost56939} и перечнем OWASP
Top~10~\cite{owasp_asvs}.

\emph{Внедрение операторов структурированного языка запросов
(Structured Query Language, SQL) через входные данные.} Все запросы
к базе данных параметризованы через драйвер pgx (использование
плейсхолдеров \$1, \$2, \dots); конкатенация строк в SQL-запросах
исключена.

\emph{Межсайтовый скриптинг (XSS).} Автоматическое экранирование
вывода во Flutter; HTML-рендеринг пользовательского контента
не используется.

\emph{Межсайтовая подделка запросов (CSRF).} Токен доступа
передаётся в заголовке Authorization (не в cookie),
что устраняет основной вектор CSRF-атак на API.

\emph{Передача токена при WebSocket-апгрейде.} Браузерное API
WebSocket~\cite{rfc6455} не позволяет задавать произвольные
HTTP-заголовки при установке соединения, поэтому токен доступа
передаётся в строке параметров запроса (?token=\dots),
как показано на Рисунке~\ref{fig:seq-join}. В качестве
альтернативы рассматривалась передача токена через расширение
Sec-WebSocket-Protocol, формально предусмотренное
для подвыбора прикладного протокола; этот подход требует
нестандартной интерпретации поля и не имеет встроенной поддержки
в используемой серверной библиотеке gorilla/websocket. Связанные
с передачей токена в параметрах запроса риски (попадание токена
в журналы доступа обратного прокси, кешируемые URL у промежуточных
узлов) компенсируются короткой длительностью жизни токена доступа
(15~мин) и отключением записи параметров запросов в журналах Caddy
для эндпоинтов /ws/*.

\emph{Перехват токенов.} Канал передачи между клиентом и сервером
шифруется протоколом TLS 1.3; срок действия токена доступа ограничен
15 минутами для минимизации последствий компрометации.

\emph{Небезопасный прямой доступ к ресурсам (IDOR).} Каждый
защищённый эндпоинт проверяет не только валидность токена,
но и принадлежность ресурса запрашивающему пользователю
(сопоставление поля owner\_id ресурса с идентификатором,
извлечённым из токена).

\subsection{Модели и алгоритмы интерактивного взаимодействия}

В настоящем подразделе разработаны модели и алгоритмы,
обеспечивающие функции интерактивного взаимодействия с обучающимися
(см.~подраздел~2.1).

\subsubsection{Модель интерактивной учебной сессии}

Под интерактивной учебной сессией в разработанной модели понимается
ограниченный по времени сеанс коллективной работы преподавателя
и группы обучающихся, в ходе которого преподаватель предъявляет
последовательность вопросов из выбранного банка, а участники
предоставляют ответы через клиентское приложение.

Состояние сессии представлено конечным автоматом
$M = (S, S_0, F, T)$, где $S$~--- множество состояний;
$S_0 \in S$~--- начальное состояние; $F \subseteq S$~--- множество
финальных состояний; $T$~--- множество переходов вида
$(s, e, s')$, где $s, s' \in S$~--- исходное и целевое состояния,
$e$~--- событие, инициирующее переход. Множество состояний $S$
образовано пятью элементами: \emph{created}, \emph{waiting},
\emph{active}, \emph{review} и \emph{finished}; начальное
состояние $S_0 = \emph{created}$, множество финальных состояний
$F = \{\emph{finished}\}$. Множество переходов $T$ задано
диаграммой состояний на Рисунке~\ref{fig:state-session}.

Семантика состояний: \emph{created}~--- сессия создана
преподавателем, числовой код выпущен, подключение участников
ещё не открыто; \emph{waiting}~--- открыто подключение
участников, активный вопрос отсутствует; \emph{active}~---
участникам предъявлен вопрос, ожидаются ответы; \emph{review}~---
приём ответов на текущий вопрос завершён, участникам отображается
правильный ответ и локальная статистика; \emph{finished}~---
сессия завершена, результаты зафиксированы в базе данных.
Диаграмма состояний сессии и переходов между ними приведена
на Рисунке~\ref{fig:state-session}.

В модели предусмотрены три режима проведения сессии, различающихся
способом синхронизации участников. \emph{Classic}~--- синхронный
групповой режим, в котором все участники одновременно отвечают
на общий текущий вопрос; темп задаёт преподаватель. Применим
для семинарских занятий с обсуждением. \emph{Timer}~---
индивидуальный режим с ограничением времени, в котором каждый
участник проходит вопросы в собственном темпе с таймером на каждый
вопрос. Применим для контрольного тестирования.
\emph{Race}~--- индивидуальный соревновательный режим без
ограничения времени с определением победителя~--- участника,
первым правильно ответившего на все вопросы.

Состояние сессии хранится на стороне сервера и является единым
источником истины; обновления транслируются клиентам по WebSocket.
Формальное определение конечного автомата $M$ используется при
обработке временного обрыва WebSocket-соединения. Клиент
переподключается, считывает из базы данных текущее состояние
$s \in S$ и переходит в него независимо от состояния до обрыва.
Такое восстановление корректно: чтение состояния из БД эквивалентно
присваиванию $s' = s$ и не зависит от истории переходов
до обрыва.

Динамика взаимодействия клиентских приложений преподавателя

\subsubsection{Модель игровой прогрессии}

Модель игровой прогрессии направлена на поддержание устойчивой
учебной мотивации обучающихся и повышение регулярности обращения
к учебному материалу за счёт механизмов внешней
мотивации~\cite{deterding2011,sailer2020}. Разработанная модель
включает четыре взаимосвязанных механизма: накопление очков опыта
(XP), систему уровней с замедляющимся ростом, систему достижений
и учёт серий активных дней. Соотнесение
с базовыми психологическими потребностями по теории
самодетерминации Деси и Райана~\cite{deci_ryan_2000} выглядит
следующим образом: накопление очков опыта и переход на следующий
уровень обеспечивают чувство компетентности через измеримый
прогресс; индивидуальный темп прохождения вопросов в режимах
timer и race поддерживает потребность обучающегося в автономии;
таблица лидеров и групповая сессия в режиме classic~--- потребность
в связности с другими участниками учебной группы.

Начисление очков опыта за ответ на вопрос в ходе интерактивной
сессии вычисляется по формуле~\eqref{eq:xp}.
\begin{equation}\label{eq:xp}
    \Delta XP = w_c \cdot d \cdot \left(1 - \beta \cdot \min\!\left(\frac{t}{T},\,1\right)\right) \cdot 10,
\end{equation}
где $w_c$~--- множитель за корректность ответа ($w_c = 1$
для правильного, $w_c = 0$ для неправильного); $d$~--- оценка
сложности вопроса (целое число от 1 до 5); $t$~--- время,
затраченное на ответ; $T$~--- максимальное время на ответ;
$\beta = 0{,}5$~--- коэффициент штрафа за продолжительность
ответа. Значение $\beta = 0{,}5$ принято по соображениям
симметрии: при таком значении быстрый верный ответ ($t \to 0$)
начисляет вдвое больше очков, чем верный ответ на границе
таймера ($t = T$). Систематическая количественная калибровка
параметра по данным реального учебного процесса в работе
не проводилась и относится к перспективам дальнейшего исследования
(см.~подраздел~3.8). Ограничение отношения $t/T$ сверху единицей исключает
отрицательные значения $\Delta XP$ при ответах за пределами
лимита времени. При проведении сессии в режиме без ограничения
времени на вопрос ($T = 0$) коэффициент скорости в выражении
$1 - \beta \cdot \min(t/T, 1)$ принимается равным $1$, что
обеспечивает корректное значение $\Delta XP$ независимо от времени
ответа. При неверном ответе ($w_c = 0$) очки опыта не начисляются
независимо от затраченного времени; частичный учёт скорости
неверного ответа не предусмотрен, поскольку это противоречит
назначению механики поощрения. Множитель $10$ согласован
с пороговыми значениями системы уровней~\eqref{eq:level}:
при коэффициенте уровня $k_L = 100$ переход на первый уровень
требует накопления $100$~единиц очков опыта, что соответствует
примерно десяти правильным ответам на вопросы средней сложности
$d = 2$~--~$3$. Значение $k_L = 100$ принято исходя из этого
ориентира; количественная калибровка по данным реального учебного
процесса в работе не проводилась.

Перевод накопленных очков опыта в уровень $L$ выполняется
по формуле~\eqref{eq:level}.
\begin{equation}\label{eq:level}
    L = \left\lfloor \sqrt{XP / k_L} \right\rfloor, \quad k_L = 100.
\end{equation}
Квадратичная зависимость $XP(L) = k_L \cdot L^2$ обеспечивает
быстрый рост уровня в начале использования сервиса и замедление
по мере накопления опыта, моделируя убывающую отдачу
от дополнительной активности~\cite{sailer2020}.

Система достижений реализована как множество однократных наград
за выполнение определённых условий: получение первого правильного
ответа, прохождение сессии без ошибок, накопление установленного
объёма очков опыта, серия активных дней заданной длины.
Идемпотентность операции выдачи достижения обеспечивается
уникальным ключом (user\_id, badge\_code, course\_id)
в таблице выданных достижений. Учёт серий активных дней мотивирует
обучающихся к регулярному обращению к сервису между занятиями.

\subsubsection{Алгоритм индивидуального интервального повторения}

Для индивидуального повторения учебного материала между сессиями
использован алгоритм SuperMemo SM-2~\cite{wozniak1990}, эффективность
которого в массовых системах распределённого обучения подтверждена
эмпирически~\cite{alyshova2022}.

Каждой паре «обучающийся, вопрос» сопоставляется состояние повторения
$\sigma = (EF, I, n)$, где $EF$~--- коэффициент лёгкости вопроса;
$I$~--- текущий интервал повторения в сутках; $n$~--- число
последовательных успешных повторений. Начальное состояние:
$EF_0 = 2{,}5$, $I_0 = 0$, $n_0 = 0$.

Обновление состояния после ответа обучающегося с оценкой качества
$q \in \{0, \ldots, 5\}$ выполняется по правилам~\eqref{eq:sm2-update}.
\begin{equation}\label{eq:sm2-update}
    \begin{cases}
        \text{если } q < 3: & n \leftarrow 0, \quad I \leftarrow 1; \\
        \text{если } q \geq 3 \text{ и } n = 0: & I \leftarrow 1, \quad n \leftarrow n + 1; \\
        \text{если } q \geq 3 \text{ и } n = 1: & I \leftarrow 6, \quad n \leftarrow n + 1; \\
        \text{если } q \geq 3 \text{ и } n > 1: & I \leftarrow \lceil I \cdot EF \rceil, \quad n \leftarrow n + 1.
    \end{cases}
\end{equation}
В оригинальной формулировке алгоритма~\cite{wozniak1990} обновление
интервала повторения определяется выражением~\eqref{eq:sm2-orig}:
\begin{equation}\label{eq:sm2-orig}
    I(n) = I(n-1) \cdot EF,
\end{equation}
без округления. В разрабатываемой системе интервал повторения
хранится с точностью до суток как естественная единица планирования
учебной активности; округление вверх $\lceil I \cdot EF \rceil$
в правиле~\eqref{eq:sm2-update} выбрано для гарантии минимального
ненулевого шага роста интервала при значениях $EF \cdot I$,
близких к целому.

Коэффициент лёгкости $EF$ обновляется по формуле~\eqref{eq:sm2-ef}.
\begin{equation}\label{eq:sm2-ef}
    EF \leftarrow \max\bigl(1{,}3,\; EF + 0{,}1 - (5 - q) \cdot (0{,}08 + (5 - q) \cdot 0{,}02)\bigr).
\end{equation}
Нижняя граница $EF \geq 1{,}3$ установлена в оригинальной
формулировке алгоритма~\cite{wozniak1990} и предотвращает
неограниченное сжатие интервала повторения для вопросов
с устойчиво низкой оценкой качества ответа: при отсутствии
ограничения коэффициент лёгкости стремился бы к нулю,
а интервал~--- к одному дню, что приводило бы к чрезмерно
частому предъявлению карточки. Дата следующего предъявления
карточки определяется как $d_{\text{next}} = d_{\text{cur}} + I$.

В классической реализации SM-2 оценка качества ответа $q$
запрашивается у обучающегося явно после предъявления правильного
варианта. В разрабатываемом сервисе оценка $q$ формируется
автоматически по корректности ответа и времени реакции
по правилу~\eqref{eq:q-score}.
\begin{equation}\label{eq:q-score}
    q =
    \begin{cases}
        5, & \text{ответ верный и } t \leq T/2; \\
        4, & \text{ответ верный и } T/2 < t \leq T; \\
        3, & \text{ответ верный и } t > T; \\
        2, & \text{ответ неверный и } t < T; \\
        1, & \text{ответ неверный, таймер истёк.}
    \end{cases}
\end{equation}
Интерпретация границы $q = 1$ и $q = 2$ представляет собой
осознанное отклонение от шкалы Возняка~\cite{wozniak1990}, в которой
$q = 1$ соответствует субъективной оценке «ответ показался
незнакомым», а $q = 2$~--- «неверный ответ при последующем
узнавании показанного варианта». В автоматическом режиме явной
субъективной оценки нет, поэтому таймаут (отсутствие активной
попытки выбора варианта) трактуется как более глубокое незнание,
чем активный неверный ответ обучающегося. При этом таймаут
по UI отличается от завершения сессии по обрыву соединения:
в правиле~\eqref{eq:q-score} учитываются только попытки, в которых
клиент успешно установил WebSocket-канал и предъявил вопрос
обучающемуся; ответы, не зафиксированные сервером из-за обрыва
соединения, не попадают в обновление состояния SM-2.
При проведении сессии в режиме без ограничения времени на вопрос
($T = 0$) правило~\eqref{eq:q-score} не применимо: для верного
ответа принимается $q = 4$, для неверного~--- $q = 2$. Уровень
$q = 0$ из исходной шкалы SuperMemo не используется, поскольку
требует от обучающегося явной оценки «полное непонимание», которая
несовместима с автоматическим формированием $q$. Коэффициент
лёгкости $EF$ обновляется по формуле~\eqref{eq:sm2-ef} после
каждого ответа, в том числе при $q < 3$, что соответствует
оригинальному алгоритму~\cite{wozniak1990}; обновление $EF$
применяется после обновления интервала повторения по
правилу~\eqref{eq:sm2-update}.

Автоматическое формирование оценки $q$ устраняет необходимость
отдельного интерфейса самооценки и обеспечивает интеграцию
интервального повторения в цикл интерактивной учебной сессии:
после каждого ответа в сессии помимо начисления очков опыта
вызывается обновление состояния SM-2, что обеспечивает естественную
аккумуляцию учебного материала~--- один раз отвеченный вопрос
автоматически попадает в очередь повторения обучающегося.

\subsubsection{Модель оценки качества тестового инструмента}

Для оценки качества банка вопросов как измерительного инструмента
применяются психометрические показатели классической теории
тестов~\cite{cronbach1951,crocker_algina}.

Коэффициент $\alpha$ Кронбаха оценивает внутреннюю согласованность
тестового инструмента~--- тенденцию всех пунктов измерять одну
и ту же латентную способность; рассчитывается
по формуле~\eqref{eq:cronbach}.
\begin{equation}\label{eq:cronbach}
    \alpha = \frac{k}{k - 1} \left(1 - \frac{\sum_{i=1}^{k} \sigma_{Y_i}^2}{\sigma_X^2}\right),
\end{equation}
где $k$~--- число пунктов; $\sigma_{Y_i}^2$~--- дисперсия $i$-го
пункта; $\sigma_X^2$~--- дисперсия суммарного балла. Для
дихотомических (двоичных по корректности) пунктов
$\sigma_{Y_i}^2 = p_i (1 - p_i)$, и формула~\eqref{eq:cronbach}
совпадает с коэффициентом надёжности
Кьюдера--Ричардсона KR-20~\cite{crocker_algina}. В качестве
ориентира интерпретации в работе принят порог
$\alpha \geq 0{,}7$~\cite{crocker_algina}, восходящий
к рекомендации Нанналли для тестовых инструментов умеренной
длины. Указанный порог чувствителен к числу пунктов
тестового инструмента: при $k < 10$ значение $\alpha$
систематически занижается, и достижение порога 0,7
не всегда возможно без увеличения длины теста.

Индекс трудности $p_i$ есть доля правильных ответов на $i$-й
вопрос по выборке участников. Точечно-бисериальная корреляция
ответа на пункт с суммарным баллом по тесту рассчитывается
по формуле~\eqref{eq:rpb}.
\begin{equation}\label{eq:rpb}
    r_{pb,i} = \frac{\bar{X}_{+} - \bar{X}_{-}}{\sigma_X} \sqrt{p_i (1 - p_i)},
\end{equation}
где $\bar{X}_{+}$ и $\bar{X}_{-}$~--- средние суммарные баллы
участников, соответственно правильно и неправильно ответивших
на $i$-й пункт. Положительные значения $r_{pb,i}$ свидетельствуют
о работающем пункте, отрицательные~--- о кандидате на исключение
или переформулировку.

Принятые в работе ориентиры интерпретации показателей соответствуют
рекомендациям классической теории тестов~\cite{crocker_algina}:
$p_i \geq 0{,}85$~--- пункт слишком лёгкий; $p_i < 0{,}30$~---
слишком сложный; $r_{pb,i} \leq 0{,}2$~--- низкая дискриминативная
способность пункта. Помимо длины тестового инструмента, на устойчивость
оценки коэффициента $\alpha$ влияет размер выборки участников;
рекомендации по интерпретации показателей при малых выборках
рассмотрены в~подразделе~2.6.

Содержательным результатом разработанной модели является изменение
рабочего процесса преподавателя: итеративное улучшение банка
тестовых вопросов становится частью цикла учебной сессии вместо
отдельного процесса постфактумного статистического анализа
в специализированном программном обеспечении (SPSS, R, Jamovi).
Сравнительный анализ существующих платформ интерактивного обучения
(см.~подраздел~1.1.4) показал, что ни одна из рассмотренных
платформ не предоставляет психометрической аналитики банка
тестовых вопросов в рабочем месте преподавателя, что подтверждает
содержательную новизну принятого архитектурного решения.

\subsubsection{ER-диаграмма базы данных}

База данных сервиса спроектирована как реляционная схема
с табличной частью в третьей нормальной форме~\cite{kleppmann2017}
и полуструктурированной частью~--- варианты ответа на вопросы
хранятся в полях типа JSONB. Полуструктурированные поля обеспечивают
гибкость при появлении новых типов вопросов без миграции схемы.
Помимо этого, в схему включён денормализованный агрегат
user\_progress, обоснование которого приведено в~2.5.2.

сессия как сущность модели данных хранится в таблице
rooms; термины «сессия» и «комната» (room) применительно
к экземпляру таблицы используются взаимозаменяемо.

Полные определения таблиц и индексов содержатся в файлах миграций
migrations/0001\_init~--- migrations/0010. Ключевые
конструктивные решения модели данных описаны ниже.

Уникальный составной ключ
$UNIQUE (room\_id, question\_id, participant\_id)$
на таблице answers предотвращает повторную регистрацию
ответа одного участника на один и тот же вопрос в рамках одной
сессии и обеспечивает устойчивость системы к повторным отправкам,
вызванным двойным нажатием или сетевыми сбоями.

Тип вопроса описывается перечислением (ENUM) question\_kind
со значениями single\_choice, multi\_choice,
open\_text, rating, qna, true\_false
и numerical. Варианты ответа и правильный ответ хранятся
в JSONB-полях, что обеспечивает гибкость при появлении новых типов
вопросов без изменения схемы таблицы.

Состояние интервального повторения для пары
$(user\_id, question\_id)$ хранится в таблице
sm2\_states с полями ef REAL, interval\_days
INTEGER, repetitions INTEGER, due\_date DATE.
Индекс $idx\_sm2\_due (user\_id, due\_date)$
обеспечивает быструю выборку карточек, запланированных на текущую
дату, при росте банка SM-2-состояний до десятков тысяч записей.

Денормализованный агрегат user\_progress с первичным ключом
$(user\_id, course\_id)$ хранит общие очки опыта, уровень и серию
активных дней обучающегося в курсе. Решение о хранении агрегата
принято исходя из частоты чтения таблицы лидеров в ходе
интерактивной сессии: каждое сообщение WebSocket-канала о смене
лидера требует чтения текущих значений XP по всем участникам, что
при вычислении агрегата по требованию приводило бы к индексному
range-scan для каждого участника на каждое сообщение. Хранение
готового агрегата устраняет дополнительный запрос к таблице
answers и упрощает обработчик. Согласованность агрегата
с журналом начислений ответов (таблица answers, поле
awarded\_xp) обеспечивается одновременным обновлением
обеих таблиц в одной транзакции; возможная конкуренция строк
user\_progress при параллельных ответах одного обучающегося
не наблюдалась в нагрузочном тестировании в силу низкой частоты
ответов одного участника (не чаще одного раза в несколько секунд).

\subsection{Ограничения принятых моделей}

Разработанные в главе модели и алгоритмы построены с рядом
осознанных упрощений, которые рассматриваются ниже как
ограничения, подлежащие учёту при интерпретации результатов
и формирующие направления дальнейшего исследования.

\emph{Правило автоматического оценивания $q$ (формула~\eqref{eq:q-score}).}
Правило отображает наблюдаемую пару «корректность~--- латентность»
в шкалу $\{1, \ldots, 5\}$ алгоритма SM-2, отклоняясь
от субъективной шкалы Возняка в части различения уровней $q = 1$
и $q = 2$. Эвристический характер правила и фиксированные границы
интервалов времени делают модель чувствительной к индивидуальному
темпу ответа обучающегося; количественная калибровка границ
по данным реального учебного процесса в работе не проводилась.

\emph{Формула начисления очков опыта~\eqref{eq:xp}.} Значения
параметров $\beta = 0{,}5$ и $k_L = 100$ приняты по соображениям
симметрии и согласованности с темпом учебной сессии (см.~2.4.2).
Систематическая оценка чувствительности распределения $\Delta XP$
и темпа роста уровня к выбору параметров в диапазоне
$\beta \in [0{,}3;\,0{,}7]$ и $k_L \in [50;\,200]$ в работе
не выполнена.

\emph{Конечный автомат учебной сессии (см.~Рис.~\ref{fig:state-session}).}
Модель состояний не содержит состояния \emph{paused}: преподаватель
не может приостановить активную сессию с сохранением состояния
участников и возобновить её позднее. Расширение автомата на состояние
паузы относится к перспективам развития.

\emph{Ролевая модель пользователей.} В модели выделены три плоские
роли (преподаватель, обучающийся, администратор) без иерархии
курсов и кафедр; роль «куратора курса», агрегирующего права
нескольких преподавателей, в работе не моделируется.

\emph{Модель оценки качества тестового инструмента.}
Классическая теория тестов предполагает однородную выборку
участников и независимость пунктов; для коротких тестовых
инструментов из $k < 10$ пунктов оценка коэффициента $\alpha$
Кронбаха систематически занижается. Альтернативные модели
психометрики на основе теории отклика на пункт (Item Response
Theory) обладают меньшей чувствительностью к длине теста, но
требуют большего объёма данных для оценки параметров пунктов;
их применение отнесено к перспективам исследования
(см.~подраздел~3.8).

\subsection{Выводы по главе 2}

В главе обоснован выбор монолитной клиент-серверной архитектуры
с открытым программным интерфейсом и определён технологический
стек: Go~1.25 с фреймворком Gin для серверной части, PostgreSQL~16,
Redis~7, обратный прокси-сервер Caddy~2 для серверной инфраструктуры,
Flutter~3.35 для кроссплатформенного клиента. Спроектирована
ролевая модель пользователей с разграничением доступа по схеме
RBAC, дополненной проверкой владения ресурсом, и реализована
JWT-аутентификация с ротацией рефреш-токенов; меры защиты
определены по требованиям ГОСТ Р 56939-2024 и перечню OWASP
Top~10. Разработаны модель интерактивной учебной сессии в виде
конечного автомата с тремя режимами проведения, модель игровой
прогрессии, алгоритм SuperMemo SM-2 с автоматическим формированием
оценки качества ответа из пары «корректность~--- время» и модель
оценки качества тестового инструмента на базе классической теории
тестов. Спроектирована реляционная модель данных в третьей
нормальной форме с целенаправленным применением полей JSONB
и денормализации агрегата игровой прогрессии. Разработанные
модели и архитектурные решения составляют основу для программной
реализации сервиса, рассматриваемой в главе~3.
