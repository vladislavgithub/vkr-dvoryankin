\section[ПРОГРАММНАЯ РЕАЛИЗАЦИЯ И РЕЗУЛЬТАТЫ]{Программная реализация и результаты}
\thispagestyle{plain}

В настоящей главе описаны программная реализация сервиса EduQuiz
в соответствии с моделями и архитектурой, разработанными в~главе~2,
инфраструктура развёртывания, оценка соответствия реализации
функциональным и нефункциональным требованиям из~подраздела~1.3.3,
результаты экспериментальной оценки производительности методом
нагрузочного тестирования, функциональное тестирование сервиса
и перспективы дальнейшей работы. Полный исходный код реализованного
сервиса опубликован в двух репозиториях системы контроля версий:
eduquiz-backend (серверная часть, язык Go) и
eduquiz-flutter (кроссплатформенный клиент, Flutter
и Dart); фрагменты реализации ключевых компонентов приведены
в~Приложении~А, скриншоты пользовательских интерфейсов~--- в~Приложении~Б.

\subsection{Архитектура реализации}

Реализация выполнена в соответствии с архитектурным решением,
обоснованным в~подразделе~2.2, и включает три самостоятельные
программные части: серверное приложение, клиентское приложение
и инфраструктурные компоненты системы хранения и проксирования.
Серверное приложение реализовано на языке Go версии~1.25 с
веб-фреймворком Gin~\cite{gin_docs}; клиентское приложение~--- на
платформе Flutter версии~3.35 с языком программирования
Dart~\cite{flutter_docs}; в качестве СУБД использована PostgreSQL
версии~16~\cite{postgresql_docs}, в качестве сервера кеширования~---
Redis версии~7. Количественные характеристики реализованной кодовой

Серверное приложение взаимодействует с клиентским через два
канала: синхронный программный интерфейс приложения (Application
Programming Interface, API) архитектурного стиля Representational
State Transfer (REST) поверх HTTP/1.1 с шифрованием Transport
Layer Security (TLS) версии~1.3 для запросов управления
(создание сессии, отправка ответа, получение состояния)
и асинхронный канал событий поверх протокола WebSocket~\cite{rfc6455}
для трансляции обновлений состояния интерактивной сессии в режиме
реального времени. Авторизация запросов выполняется по подписанному
токену JSON Web Token~\cite{rfc7519}, передаваемому в заголовке
Authorization для REST-запросов и в параметре строки
запроса при установлении WebSocket-соединения. Состояние интерактивной
сессии хранится в реляционной базе данных PostgreSQL и является
единым источником истины при восстановлении клиентов после
временного обрыва соединения (см.~подраздел~2.4.1).

Принципы разделения слоёв соответствуют шаблону «обработчик~---

обеспечивает возможность модульного тестирования бизнес-логики
без подъёма базы данных и упрощает изоляцию инфраструктурных
зависимостей.

\subsubsection{Модуль аутентификации}

Модуль аутентификации реализован в пакете internal/auth
и обеспечивает регистрацию пользователей, вход в систему,
обновление пары токенов доступа и обновления, а также авторизацию
запросов по подписанному токену. Хеширование паролей выполнено
алгоритмом bcrypt с рабочим параметром $\text{cost} = 11$, что
соответствует рекомендации OWASP Password Storage Cheat Sheet
2024~\cite{owasp_asvs} и обеспечивает время вычисления хеша
порядка 100~мс на использованной аппаратной конфигурации.
Проверка силы пароля выделена в отдельный модуль
internal/auth/password\_strength.go: проверяется длина
не менее 8 символов, наличие буквенных и цифровых символов,
отсутствие пароля в коротком списке наиболее распространённых
комбинаций.

Подпись токенов доступа и обновления выполняется алгоритмом
HS256 (HMAC SHA-256) с серверным секретом, что для монолитной
архитектуры с единым доверенным контуром предпочтительнее
асимметричного RS256 ввиду отсутствия необходимости управления
парой ключей. Токен доступа имеет срок действия 15~минут и
содержит идентификатор пользователя и его роль; проверка подписи
выполняется обработчиком запроса локально без обращения к серверу
аутентификации. Токен обновления хранится на стороне сервера
в таблице сессий с возможностью отзыва: при компрометации одного
из устройств администратор отзывает все активные сессии пользователя.

Реализованная схема является \emph{гибридной}~--- stateless
по токену доступа и stateful по токену обновления.

Для защиты от атак методом полного перебора перед эндпоинтами
аутентификации установлен ограничитель скорости запросов по
IP-адресу (token-bucket, не более 20 запросов в минуту с burst-окном
до 10 запросов), реализованный в internal/server/ratelimit.go.
Ограничитель функционирует в одной памяти и сохраняет состояние
до перезапуска приложения; при горизонтальном масштабировании
переход на разделяемое хранилище состояния отнесён к перспективам
развития (см.~подраздел~3.8).

\subsubsection{Управление учебными сущностями}

Обработчики управления учебными сущностями~--- курсами, банками
вопросов и отдельными вопросами~--- реализованы в пакете
internal/handlers/courses.go (672 строк кода) и его репозитарных
зависимостях. Для каждой сущности реализован полный набор операций
жизненного цикла: создание, чтение, обновление, удаление. Доступ
к сущностям ограничен принадлежностью преподавателю-владельцу: при
обработке запроса проверяется совпадение поля owner\_id
сущности с идентификатором пользователя, извлечённым из токена
доступа. Эта проверка дополняет ограничения целостности базы
данных и снижает риск ошибки доступа на уровне приложения.

Тип вопроса описывается перечислением question\_kind со
значениями single\_choice, multi\_choice,
open\_text, true\_false, numerical,
rating и qna. Варианты ответа и правильный ответ
хранятся в JSONB-поле options как полуструктурированные
данные, что позволяет добавлять новые типы вопросов без миграции
схемы таблицы.

Для каждого вопроса предусмотрено хранение изображения, прикрепляемого
к условию: изображение загружается через эндпоинт
POST /uploads (с ограничением размера до 5~МиБ),
сохраняется на сервере и связывается с вопросом по URL-полю.
Поддерживаются также URL-ссылки на внешние изображения для
случаев импорта банка вопросов из внешних источников.

\subsubsection{Импорт банка вопросов}

Импорт банка вопросов из внешних форматов реализован в пакете
internal/services/import. Поддерживаются два формата:
Moodle XML (файл moodle.go, 373 строки кода) и Moodle GIFT
(файл gift.go, 338 строк кода), описанные в документации
системы дистанционного обучения Moodle и используемые в системе
дистанционного обучения университета. Парсеры приводят входные
данные к единому представлению ParsedQuestion, после
чего полученные вопросы добавляются в банк через стандартный
обработчик создания вопроса.

При обработке Moodle XML применена защита от атак типа XXE
(XML External Entity) и Billion Laughs: перед парсингом из входного
документа удаляются объявления <!DOCTYPE> и определения
сущностей, что устраняет возможность подключения внешних ресурсов
или экспоненциального раскрытия вложенных сущностей. Также
выполняется ограничение размера входного документа и числа
вложенных элементов.

\subsubsection{Реактивный хаб событий WebSocket}

Канал передачи событий учебной сессии в реальном времени реализован
в пакете internal/ws (350 строк кода) по реактивной модели
с единственной горутиной-управителем (хабом). Хаб владеет
ассоциативной таблицей подписок rooms[UUID] → Set[Client]
и обрабатывает три канала команд: регистрации клиента
(register), отключения (unregister) и рассылки
сообщения подписчикам (broadcast). Все операции изменения
состояния выполняются последовательно одной горутиной без явных
блокировок (мьютексов), что упрощает рассуждения о корректности
и устраняет риск гонок данных.

Каждому подключённому клиенту соответствует отдельная горутина-писатель
с буферизованным каналом отправки сообщений ограниченной ёмкости.
При рассылке хаб неблокирующим образом помещает сообщение в канал
каждого клиента; если буфер заполнен (клиент не успевает читать
из канала), клиент отключается от хаба, чтобы медленное соединение
одного участника не задерживало рассылку остальным. Описанный
приём обеспечивает устойчивость рассылки в условиях разнородных
сетевых задержек между участниками одной учебной группы.

Типы событий, транслируемых через канал, перечислены
в~табл.~\ref{tab:ws-events}.

Подключение клиента к каналу выполняется через эндпоинт
GET /ws/rooms/:id?token=... с протокольным апгрейдом
из HTTP в WebSocket в соответствии с RFC~6455~\cite{rfc6455}.
Токен доступа передаётся в параметре строки запроса, что
обусловлено отсутствием в браузерном API WebSocket возможности
задавать произвольные HTTP-заголовки при установлении соединения
(см.~обоснование выбора в подразделе~2.3.3). При обрыве
соединения клиент выполняет повторное подключение и считывает
актуальное состояние сессии из базы данных, что обеспечивает
корректное восстановление при кратковременных сетевых сбоях
без потери прогресса учебной сессии.

\subsubsection{Реализация алгоритма SM-2}

Алгоритм индивидуального интервального повторения SM-2 (модель
из~подраздела~2.4.3) реализован в пакете internal/services
(файл sm2.go, 89 строк кода). Сигнатура основной функции:

\begin{quote}
func Sm2Update(s Sm2State, q int, now time.Time) Sm2State
\end{quote}

Функция принимает текущее состояние пары «обучающийся,~вопрос»
типа Sm2State (содержит поля EF,
IntervalDays, Repetitions, DueDate),
оценку качества ответа $q \in \{0, \ldots, 5\}$ и момент
получения ответа; возвращает обновлённое состояние согласно
правилам~\eqref{eq:sm2-update} и~\eqref{eq:sm2-ef}. Реализация
свободна от побочных эффектов и обращений к базе данных, что
обеспечивает возможность модульного тестирования без подъёма
инфраструктуры. Модульные тесты в файле sm2\_test.go
покрывают сценарии первого успешного ответа, перехода интервала
к шести суткам, сброса состояния при неуспешном ответе и
ограничения снизу коэффициента лёгкости $EF \geq 1{,}3$.

Формирование оценки качества $q$ по правилу~\eqref{eq:q-score}
выполняется в обработчике приёма ответа handlers/rooms.go
после вычисления корректности ответа и времени реакции; полученное
значение $q$ передаётся в Sm2Update, а возвращённое
состояние сохраняется в таблице sm2\_states через
репозиторий repository/sm2.go. При запросе очереди
повторения обучающимся возвращаются записи с due\_date
не позднее текущей даты, упорядоченные по приоритету.

\subsubsection{Психометрический анализатор}

Психометрический анализатор банка тестовых вопросов реализован
в файле internal/services/analytics.go (189 строк кода).
Сигнатура основной функции:

\begin{quote}
func AnalyzeBank(responses [][]int) BankAnalytics
\end{quote}

Функция принимает матрицу ответов размерности $n \times k$, где
$n$~--- число участников, $k$~--- число вопросов; элемент матрицы
содержит индикатор корректности ответа $(0$ или $1)$. Возвращается
структура BankAnalytics, содержащая коэффициент
$\alpha$~Кронбаха, индекс трудности $p_i$ и точечно-бисериальную
корреляцию $r_{pb,i}$ для каждого пункта банка, а также текстовую
интерпретацию значения $\alpha$ (по диапазонам $\geq 0{,}9$,
$\geq 0{,}8$, $\geq 0{,}7$, $\geq 0{,}6$, $< 0{,}6$) и общую
статистику (число участников, средний балл, максимально возможный
балл).

Реализация свободна от инфраструктурных зависимостей и используется
как обработчиком запроса аналитики, так и в модульных тестах.
Тесты в файле analytics\_test.go включают эталонный
пример из монографии Crocker и Algina (5~участников, 6~вопросов;
$\alpha \approx 0{,}79$)~\cite{crocker_algina}, шкалу Гуттмана
(идеальная согласованность пунктов, $\alpha \to 1$) и
вырожденный случай идентичных ответов всех участников
($\alpha = 0$). Контроль соответствия выходных значений
эталонным подтверждает корректность реализации.

\subsection{Реализация клиентского приложения}

Клиентское приложение реализовано на платформе Flutter с
поддержкой четырёх целевых платформ из единой кодовой базы:
веб-приложение (через сборку Flutter~Web), нативное приложение
iOS, нативное приложение Android и нативное приложение macOS.
Объём кода~--- 69~исходных файлов на языке Dart, организованных
в восемь функциональных модулей в директории lib/features:
auth (вход и регистрация), onboarding
(вводный сценарий), teacher (рабочее место преподавателя),
student (рабочее место обучающегося), room
(прохождение интерактивной сессии), profile (профиль
обучающегося с игровой прогрессией), repetition
(индивидуальное интервальное повторение) и admin
(административная панель).

\subsubsection{Управление состоянием и маршрутизация}

Управление состоянием приложения выполнено с использованием
библиотеки Riverpod версии~2.6~\cite{riverpod_docs} с
кодогенерацией провайдеров (riverpod\_annotation).
Каждый модуль определяет собственный набор провайдеров: провайдер
текущего состояния аутентификации, провайдер списка курсов
преподавателя, провайдер деталей выбранной сессии и так далее.
Зависимости между провайдерами выстраиваются декларативно;
обновление состояния одного провайдера автоматически инвалидирует
зависимые от него.

Маршрутизация реализована средствами библиотеки go\_router
версии~14 с декларативной защитой маршрутов по роли пользователя.
В корневом файле app\_router.dart определены маршруты
для каждого функционального модуля; для маршрутов рабочих мест
преподавателя и обучающегося установлена проверка соответствия
роли пользователя, извлечённой из токена доступа. При попытке
обращения к недоступному маршруту выполняется автоматический
переход на экран аутентификации.

\subsubsection{Адаптивная вёрстка}

Для поддержки четырёх целевых платформ с различными размерами
экрана введены пороговые значения ширины экрана: планшетный
вариант ($\geq 700$~пикселей) и десктопный
($\geq 1100$~пикселей). На основе этих порогов в экранах
применяется компоновка одной или нескольких колонок: на узких
экранах элементы отображаются последовательно, на широких~---
параллельно. Адаптивная вёрстка реализована с использованием
встроенных средств Flutter (LayoutBuilder,
MediaQuery) без подключения внешних библиотек
адаптивной разметки.

\subsubsection{Подключение к каналу событий}

Подключение к каналу событий интерактивной сессии реализовано
в модуле room через библиотеку web\_socket\_channel.
Класс RoomEventsClient устанавливает соединение по
адресу ws[s]://.../ws/rooms/:id?token=..., преобразует

обновление пользовательского интерфейса.

\subsection{Инфраструктура развёртывания}

Контейнеризация сервиса выполнена с использованием
Docker~\cite{docker_docs} и Docker~Compose. Одна композиция
описывает как development-, так и production-среду с подменой
переменных окружения. В development-среде доступны два сценария
запуска: make dev запускает PostgreSQL и Redis
в Docker-контейнерах, а серверное приложение~--- локально
командой go run; make up поднимает все сервисы
(включая серверное приложение) в контейнерах одной командой.

Production-развёртывание выполнено на виртуальной машине Yandex
Cloud с операционной системой Ubuntu Server LTS (2~vCPU, 2~ГБ RAM,
20~ГБ SSD). Обратный прокси Caddy~\cite{caddy_docs} обеспечивает
автоматический выпуск и обновление TLS-сертификата Let's Encrypt
по протоколу ACME. Деплой автоматизирован средствами GitHub
Actions: при push в ветку deploy-demo рабочий процесс
выполняет сборку Docker-образа на виртуальной машине, остановку
и запуск композиции и накат миграций схемы базы данных. Сервис
доступен по доменному имени eduquizz.ru с TLS~1.3.

\textbf{Расчётное обоснование показателей надёжности.} Архитектура
сервиса декомпозируется на пять узлов: серверное приложение,
PostgreSQL, Redis, Caddy и виртуальная машина хостинг-провайдера.
Для каждого узла приняты значения интенсивности отказов $\lambda_i$
и среднего времени восстановления $T_{\text{в}i}$ по эксплуатационным
данным аналогичных систем и соглашению об уровне обслуживания
провайдера в соответствии с подходом, изложенным в~\cite{ostreikovsky,gost27001}.
Для произвольного узла вероятность безотказной работы за время~$t$
при экспоненциальном законе вычисляется по формуле
$P_i(t) = e^{-\lambda_i t}$; средняя наработка до отказа
$\text{MTBF}_i = 1/\lambda_i$; коэффициент готовности
$k_{\text{г}\,i} = \text{MTBF}_i / (\text{MTBF}_i + T_{\text{в}\,i})$.
В одиночной конфигурации без резервирования системный коэффициент
готовности рассчитывается как произведение по всем узлам. При
типовых значениях интенсивности отказов компонентов промышленного
программного обеспечения и среднего времени восстановления
с учётом плановых перезапусков виртуальной машины, обновлений
операционной системы и миграций схемы базы данных расчётный
показатель готовности составляет величину порядка $0{,}99$, что
соответствует ожидаемому времени недоступности порядка нескольких
суток в год и удовлетворяет нефункциональному требованию~2
для одиночной конфигурации (см.~подраздел~1.3.3). Достижение
более высокого значения готовности порядка $0{,}999$, заявленного
требованием~2 в режиме с резервированием, обеспечивается горячим
резервированием критичных узлов: репликацией PostgreSQL
master-replica и горизонтальным масштабированием stateless-приложения
через балансировщик Caddy. Реализация механизмов резервирования
отнесена к перспективам развития сервиса (см.~подраздел~3.8).

сублинейно по отношению к числу обучающихся: с 12~мс при $N = 30$
до 233~мс при $N = 500$. В ходе проведённого нагрузочного тестирования
ошибки класса 5xx (внутренние ошибки сервера) не наблюдались.
Полученные значения позволяют заключить, что разработанная
архитектура в одиночной конфигурации без репликации устойчиво
обслуживает не менее 500 одновременных обучающихся в одной сессии,
что превышает реалистичный размер аудитории дисциплины
\textit{«Базы данных»} кафедры АСУ РГУ Нефти и Газа (НИУ)
имени И.М. Губкина в 5--10 раз. Тем самым выполнено
нефункциональное требование~1 (см.~подраздел~1.3.3).

Высокое значение латентности эндпоинта POST /auth/register
в первоначальной серии прогонов (приведено в табл.~\ref{tab:loadtest-results})
обусловлено вычислительной стоимостью алгоритма bcrypt с рабочим
параметром $\text{cost} = 12$, использовавшимся на момент проведения
тестирования и обеспечивавшим время вычисления хеша порядка 250~мс.
По результатам анализа нагрузочного теста рабочий параметр снижен
до $\text{cost} = 11$ (порядка 100~мс на вычисление хеша), что
соответствует актуальной рекомендации OWASP Password Storage Cheat
Sheet 2024~\cite{owasp_asvs} и приблизительно в 2,5 раза снижает
latency-вектор атаки на эндпоинт регистрации при сохранении
стойкости к brute-force. Эндпоинт регистрации не относится
к критическому пути обработки запросов сессии: регистрация
обучающегося выполняется однократно при первом обращении к сервису
и в реальной эксплуатации не совпадает по времени с моментом
начала учебной сессии.

\subsection{Перспективы дальнейшей научно-исследовательской работы}
\label{sec:future-work}

К моменту завершения работы сервис EduQuiz реализован как
работоспособный программный продукт, обеспечивающий основные
сценарии использования преподавателя и обучающегося; на его
основе формируется задел для дальнейшей научно-исследовательской
работы. Перспективы расширения сервиса и направления
научно-исследовательской работы включают следующее.

\textbf{Полная интеграция с системой дистанционного обучения
университета.} В реализованной версии загрузка банков вопросов
выполняется в форматах Moodle XML и GIFT. Полная двусторонняя
интеграция с порталом edu.gubkin (синхронизация курсов и списков
обучающихся, отправка результатов сессий в журнал) запланирована
в дорожной карте развития.

\textbf{Промышленный мониторинг и резервирование.} Подключение
системы мониторинга Prometheus с экспортом per-endpoint латентности
и доли ошибок, визуализация в Grafana и алертинг через Alertmanager.
Репликация PostgreSQL master-replica с автоматическим failover
для повышения коэффициента готовности; целевое RTO 30~секунд,
RPO 1~секунда. Переход от in-memory ограничителя скорости запросов
к распределённому ограничителю на базе Redis для возможности
горизонтального масштабирования серверного приложения.

\textbf{Адаптивное тестирование на основе теории отклика на пункт.}
Ключевая исследовательская проблема, выделенная по результатам
работы, состоит в том, что существующие интерактивные тестирующие
системы измеряют формальную правильность ответа на отдельный
вопрос, но не оценивают латентный уровень подготовки обучающегося
независимо от состава конкретного теста. Совместное моделирование
параметров банка вопросов методами теории отклика на пункт
(Item Response Theory, IRT~\cite{crocker_algina}) и оценка
латентного уровня $\hat{\theta}$ обучающегося, а также построение
адаптивного режима тестирования с подбором следующего вопроса
по критерию максимума функции информации Фишера в окрестности
текущей оценки $\hat{\theta}$, образуют содержание дальнейшей
научно-исследовательской работы. Реализованные в работе архитектура
сервиса, модель данных и психометрическая аналитика средствами
классической теории тестов образуют техническую основу для
дальнейших исследований по перечисленным направлениям.

\textbf{Оценка валидности атрибуции ответа в условиях распространения
больших языковых моделей.} Дополнительной исследовательской задачей,
получающей актуальность в условиях широкого распространения
общедоступных сервисов на основе больших языковых моделей
(ChatGPT, GigaChat, YandexGPT и др.), является разработка модели
оценки достоверности самостоятельного выполнения учебного
тестирования обучающимся. На уровне модели данных в разрабатываемом
сервисе предусмотрено хранение поведенческой телеметрии каждой
попытки ответа (время ответа, последовательность переключений
активного окна, идентификатор устройства), необходимой для
последующего построения классификатора валидности атрибуции
попытки. Разработка такого классификатора и интегральной оценки
знаний, учитывающей одновременно измерительную точность тестового
инструмента и валидность атрибуции ответа, выделена
как самостоятельное направление последующей научно-исследовательской
работы.

\textbf{Выводы по главе 3.}
В главе описана программная реализация сервиса EduQuiz~---
серверное приложение объёмом около 7500 строк кода на языке Go
(шесть пакетов внутреннего кода) и кроссплатформенное клиентское
приложение объёмом около 14\,000 строк кода на платформе Flutter
(восемь функциональных модулей). Подробно рассмотрены реализация
гибридной JWT-схемы аутентификации, реактивный WebSocket-хаб без
блокировок, импорт банка вопросов из форматов Moodle XML и GIFT
с защитой от атак типа XXE, реализация алгоритма SuperMemo SM-2
и психометрического анализатора банка вопросов средствами
классической теории тестов. Сервис развёрнут на доменном имени
eduquizz.ru в инфраструктуре Yandex Cloud с автоматическим
выпуском TLS-сертификата средствами Caddy и автоматическим
деплоем через GitHub Actions. Сопоставление с функциональными
и нефункциональными требованиями из~подраздела~1.3.3
(см.~табл.~\ref{tab:fr-implementation} и~\ref{tab:nfr-quality})
показало, что совокупность требований закрывается реализованными
компонентами. Методом нагрузочного тестирования установлено,
что разработанная архитектура устойчиво обслуживает не менее
500~одновременных обучающихся в одной сессии при
$p_{99}$~латентности критического пути не выше 233~мс и нулевой
доле ошибок класса 5xx; нефункциональное требование
к производительности выполнено. Проведено функциональное
тестирование разработанного сервиса (11~модулей тестов
критичных компонентов серверной части и 43~тест-кейса
по 9~основным сценариям сквозного использования); все тесты
пройдены без дефектов уровня \emph{blocker} и \emph{critical}.
Сформулированы перспективы дальнейшей научно-исследовательской
работы~--- задел для последующих этапов исследования.
